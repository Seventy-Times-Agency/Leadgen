# Product Readiness Audit (MVP)

Дата оценки: 2026-04-19

## Итоговая оценка

**Текущая готовность: ~65% до MVP-ready.**

Проект уже умеет пройти основной happy-path: принять запрос в Telegram, собрать лиды из Google Places, сделать enrichment (сайт + отзывы + AI), отдать карточки и Excel. Но до "завершённого продукта" не хватает production-обвязки: миграции, тесты, документация запуска, observability и эксплуатационные гарантии.

## Что уже готово (сильные стороны)

- Реализован полноценный bot-flow на aiogram с FSM и лимитами пользователя.
- Есть pipeline end-to-end: discovery → persistence → enrichment → aggregate insights → delivery.
- Реализован сбор лидов через Google Places Text Search + Place Details.
- Реализован website enrichment (контакты, соцсети, сигналы сайта).
- Есть AI-анализ одного лида и базовые инсайты по базе.
- Есть экспорт в XLSX с полями для продажной квалификации.

## Критические пробелы до "завершённого продукта"

### 1) База и миграции

- Сейчас схема создаётся через `metadata.create_all` при старте.
- Для production-нужен Alembic: версионирование схемы, повторяемый деплой, откаты.

**Что сделать:**
1. Подключить Alembic, создать initial migration.
2. Убрать implicit `create_all` из startup.
3. Добавить `make migrate` / `make upgrade` команды.

### 2) Очереди/надёжность фоновых задач

- Поиск запускается через `asyncio.create_task` внутри процесса бота.
- При рестарте процесса задачи теряются; нет ретраев/DLQ.

**Что сделать:**
1. Вынести `run_search` в worker (Celery/arq/RQ).
2. Добавить retry policy (сетевые и 5xx).
3. Добавить идемпотентность job по `query_id`.

### 3) Тестирование и quality gates

- В репозитории отсутствуют unit/integration тесты.
- Нет CI-контура, который валидирует сборку/линт/тесты.

**Что сделать:**
1. Минимум 10-15 unit-тестов (парсеры, агрегатор, filename sanitizer, JSON extraction).
2. Интеграционный тест для `run_search` с моками внешних API.
3. GitHub Actions: ruff + pytest + (опционально) mypy.

### 4) Документация запуска и env

- Нет README с пошаговым запуском и примером `.env`.
- Для нового разработчика onboarding сейчас нестабилен.

**Что сделать:**
1. Написать README (local/dev/prod, команды, архитектура).
2. Добавить `.env.example` с обязательными переменными.
3. Описать лимиты/стоимость Google Places и Anthropic.

### 5) Observability/операционный контур

- Логирование базовое, но нет метрик, health-check и алертов.
- Нет прозрачности по времени выполнения запросов, доле ошибок, стоимости enrichment.

**Что сделать:**
1. Добавить health endpoint/команду.
2. Метрики: count запросов, fail rate, p95 времени, средняя стоимость API.
3. Интеграция Sentry/OpenTelemetry.

### 6) Продуктовые/правовые элементы

- Нет пользовательских документов (privacy/terms) и UX для consent.
- Нет безопасных guardrails на потенциально рискованные сценарии рассылок.

**Что сделать:**
1. Добавить Terms + Privacy и ссылку в `/start`.
2. Политику хранения/удаления данных.
3. Ограничения на экспорт и anti-abuse правила.

## Рекомендованный план "до готового MVP" (2–3 недели)

### Неделя 1 — стабильность платформы
- Alembic + миграции.
- README + `.env.example`.
- CI с ruff/pytest.

### Неделя 2 — надёжность выполнения
- Вынести фоновые задачи в queue worker.
- Retry/idempotency.
- Базовые метрики и Sentry.

### Неделя 3 — продуктовая готовность
- UX полировка ошибок/статусов.
- Privacy/Terms + consent.
- Billing/лимиты (минимум admin-managed планы).

## Definition of Done для "MVP завершён"

Считать продукт завершённым, когда выполнены условия:

1. Реплицируемый деплой: migrations + CI green.
2. Фоновые задачи переживают рестарт и имеют retry.
3. Есть тесты на критический путь и парсинг.
4. Есть документация запуска для нового разработчика (<30 мин).
5. Ошибки и деградации наблюдаемы через метрики/алерты.
6. Есть пользовательские документы (privacy/terms) и базовые anti-abuse меры.
